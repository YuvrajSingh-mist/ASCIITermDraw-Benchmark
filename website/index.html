TermDraw-Bench

ASCII Diagram Generation and Editing Benchmark

Evaluate whether a model can actually draw in plain text. TermDraw-Bench measures box layout, topology construction, software architecture sketching, and image-conditioned ASCII editing with task directories, rendered references, structural checks, and model-specific judge prompts.

The benchmark is built as a publishable repository artifact: organized task categories, reproducible generators, Playwright rendering, Fireworks batch inference, and structured JSON judging.

Overview Quick Start

Overview

TermDraw-Bench is a benchmark for structured ASCII diagram generation and editing. Instead of grading models on prose quality alone, it checks whether they can place labels, preserve spacing, route arrows, maintain topology, and make constrained edits to an existing diagram.

The benchmark covers four families of tasks: box layout basics, network topology diagrams, diagram editing from source images, and canonical software architecture sketches. Every task ships with a prompt, a reference ASCII artifact, machine-readable L1 assertions, and a task-specific VLM judge prompt for L2 evaluation.

Category 3 is multimodal: the model receives a rendered source.png and must apply a textual edit instruction while still returning plain ASCII. The remaining categories are text-to-ASCII generation tasks.

Key Features

Benchmark Coverage

  • Box Layout Basics - simple boxes, chains, stacks, fan-in, fan-out, diamonds, and spacing-sensitive arrangements.
  • Network Topology Diagrams - brokers, parameter servers, service meshes, CDNs, DNS flows, and cluster communication patterns.
  • Diagram Editing - image-conditioned edit tasks that require preserving the original structure while applying the requested change.
  • Software Architecture Diagrams - 3-tier systems, event pipelines, API gateways, queues, ETL layouts, and other canonical engineering sketches.

Evaluation Pipeline

Two-layer scoring. L1 uses structural assertions to catch hard format and topology failures. L2 uses task-specific VLM prompts with structured JSON output to score semantic correctness more holistically.
  • Playwright PNG rendering - reference ASCII artifacts are rendered as diagram-only images for editing tasks and visual inspection.
  • Fireworks batch generation - benchmark-wide inference is wired for batch runs with reasoning disabled and network retries for transport failures.
  • Pydantic / Instructor judging - smaller synchronous judge runs can use schema-constrained JSON validation.
  • Repository-first organization - task directories, scripts, outputs, and website metadata are all arranged for extension and public contribution.

Technical Details

TermDraw-Bench is designed to be regenerated and audited from source. The task corpus is produced from generate_benchmark.py, task assets live in named category directories, and the website data is built from the repository tree rather than from hand-maintained content.

Inference and Scoring

  • Smoke generation - synchronous shared-module Fireworks calls for fast spot checks on a random subset of tasks.
  • Batch generation - dataset-backed Fireworks jobs for full-benchmark runs across all tasks.
  • L1 checker - label recall, entity count, edge presence, and format compliance derived from assertions.json.
  • L2 judge - task-specific prompts that request structured JSON outputs using Fireworks batch or Instructor-backed synchronous calls.

Assets and Rendering

  • ASCII references - each task contains a canonical reference.ascii target.
  • Source diagrams - edit tasks additionally contain source.ascii and rendered source images.
  • Diagram-only PNGs - the renderer captures the artifact itself, not the surrounding prompt or page chrome.

Publishing Notes

The website is static and GitHub Pages-ready. Its data file is generated from the real task tree, so publishing the site also publishes the benchmark structure.

Contributions are expected to happen through the repository itself: new tasks, improved judges, better renderers, and stronger evaluation baselines can all build on the same layout.
